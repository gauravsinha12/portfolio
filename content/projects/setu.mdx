## The problem

An onboarding copilot answers new-hire and new-customer questions from a body of internal docs. Plain RAG fails quietly here: when retrieval returns weakly-related chunks, the model still answers — confidently and wrong. SETU treats retrieval quality as a first-class signal. It grades what it retrieved, rewrites the question when the grade is poor, and only reaches for a live web search as a fallback before generating. It is multi-tenant, so each customer's documents stay isolated.

## Architecture

  GR -->|no| RW["Rewrite query"]
  RW --> DEC{"Improved?"}

  DEC -->|still poor| WEB["Live web search fallback"]
  WEB --> GEN
  GEN --> EVAL["Eval: context recall + faithfulness"]
`} />

The pipeline is a directed graph, not a straight line. Retrieval feeds a relevance grader; the grader's verdict decides whether to generate, rewrite and retry, or escalate to web search. Generation is followed by evaluation checks so quality is measured, not assumed.

## Engineering decisions

### LangGraph state machine vs a linear LangChain chain

A linear chain runs retrieve → generate every time, with no way to react to bad retrieval. Modeling the pipeline as a LangGraph state machine adds conditional branching — grade, then loop back to rewrite or escalate — which is exactly what "corrective" means here. The cost is real: each extra node (grading, rewriting, web search) is another model call and more latency and tokens per answer.

### Chunking strategy

Chunk size and overlap trade retrieval precision against context completeness: small chunks retrieve precisely but fragment ideas; large chunks preserve context but dilute the embedding and pull in noise. Overlap reduces boundary-splitting at the cost of duplication.

### Multi-tenant isolation in ChromaDB: collection-per-tenant vs metadata filtering

Two ways to keep tenants apart: a separate ChromaDB collection per tenant, or one shared collection with a tenant-id metadata filter on every query. Collection-per-tenant gives a hard isolation boundary and a small blast radius if something goes wrong, at the cost of more collections to manage. Metadata filtering is simpler operationally but leans on every query being filtered correctly — one missing filter leaks across tenants.

### The relevance-grading step

The grader decides whether retrieved chunks actually answer the query before the model is allowed to generate. Its inputs, its strictness, and its failure modes (false "relevant" that lets bad context through; false "irrelevant" that triggers needless rewrites and web calls) shape the whole pipeline's behavior.
