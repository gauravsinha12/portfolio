## What this tests

The interesting number for a real-time server is not "does it work" — it is the point at which added connections stop being free. This test ramps concurrent WebSocket connections against the Go server with k6 and watches four things as load climbs: end-to-end message latency, resident memory, live goroutine count, and error rate. The goal is the knee of the curve — where latency or errors start climbing faster than load.

## Interpretation

Latency stays flat and low from 100 up to about 5,000 connections — p95 holds under 60 ms — and then bends sharply at 10,000, where p95 crosses 170 ms as the single instance saturates. Memory grows close to linearly with connection count, roughly 40 KB per connection once you include the goroutine stack and per-connection I/O buffers, which is the expected cost of the goroutine-per-connection model. Goroutine count tracks at roughly 2× the connection count (a reader and a writer per connection), and the error rate stays at 0% until the knee.

On this instance size I'd cite **~5,000 concurrent connections** as the comfortable ceiling and treat 10,000 as the degradation point rather than a hard limit.

### What these numbers do not prove

This is one instance size, one message pattern (a 1 KB frame every 2 s), and a single region. It does not model real match-traffic bursts, a multi-instance deployment behind a load balancer, or adversarial reconnect storms — each of which would shift the ceiling. It also measures a synthetic echo/broadcast load, not the full match engine under gameplay.
