## What this tests

Corrective RAG grades retrieval, rewrites weak queries, and falls back to web search — all of which cost latency and tokens. This test asks whether that spend actually buys better answers. The same query set runs through both pipelines; I measure two quality signals (retrieval precision and answer faithfulness) and the two costs (added latency and cost per query). A quality win only counts if it is worth the overhead.

## Interpretation

Corrective RAG improves both quality signals — retrieval precision from **0.63 to 0.82** and faithfulness from **0.71 to 0.91** — because the grade-and-rewrite loop discards weak chunks and reformulates poor queries before generation instead of answering from bad context. The cost is real: median latency roughly **triples** (1.18 s → 3.42 s) and cost per query rises about **2.7×**, driven by the extra grading, rewriting, and occasional web-search calls.

For a correctness-sensitive onboarding assistant, where a confidently wrong answer is worse than a slow one, that trade is worth it. For latency-sensitive or very high-volume paths it may not be — which is an argument for applying the correction loop selectively rather than to every query.

### What these numbers do not prove

Results are specific to this 50-question set, this document corpus, and these models. They don't show that corrective RAG wins on every domain, and the faithfulness score is only as trustworthy as the LLM judge producing it — a weak judge can flatter either pipeline.
